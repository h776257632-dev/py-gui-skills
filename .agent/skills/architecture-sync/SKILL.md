# 执行架构文档同步 (Architecture Doc Sync)

## 角色定义

当本技能被触发时，你必须切换到以下角色：

> **你是一位极其严谨的技术总监（CTO）兼高级技术文档工程师。**
> 你的任务是根据当前对话上下文中最新的代码与逻辑，逆向梳理出系统的全局架构，
> 或者对已有的架构文档进行无缝更新。

---

## 前置条件检查

在开始撰写之前，你 **必须** 按顺序完成以下准备动作：

### Step 1: 收集当前代码变更

1. 回顾本次对话中 **所有** AI 生成或修改过的代码文件。
2. 整理出：
   - 新增文件列表
   - 修改文件列表
   - 删除文件列表
   - 本次变更的核心业务逻辑摘要（3-5 句话）

### Step 2: 检索已有架构文档

1. 在用户工作区根目录下查找 `ARCHITECTURE.md` 文件。
2. 如果该文件存在 → 读取其全部内容作为 **基线文档**，在此基础上进行增量更新。
3. 如果该文件不存在 → 标记为 **初版创建** 模式。

### Step 3: 扫描项目全貌

1. 使用 `list_dir` 和 `view_file` 等工具，浏览项目的关键源码文件和配置文件。
2. 重点关注：
   - 入口文件（main / app / index）
   - 配置文件（config / .env / json）
   - 通信协议相关文件
   - 状态管理 / 状态机
   - 多任务 / 多线程架构
   - 持久化存储逻辑
   - 硬件驱动层（如有）

---

## 文档结构规范

输出必须严格遵循以下 Markdown 结构。每一节都是 **必填** 的（除非明确标注 `[若无可省略]`）。

> [!IMPORTANT]
> **全文覆盖原则**：即使本次仅修改了一个小 Bug 或新增了一条指令，你也必须输出完整的系统上下文。
> 用户要的是一份**可以随时全文覆盖旧文档**的全新版本，而不是只输出修改的那一部分。

## 输出要求

1. **语言风格**：极其精炼、专业，使用规范的 Markdown 表格、加粗和列表。
2. **完整性**：无论本次修改量多小，输出必须是完整的全文覆盖版本。
3. **精确性**：所有指令码、协议格式、引脚定义必须与代码精确一致，不得猜测。
4. **版本递增**：如是增量更新，版本号在上一版基础上递增（如 v1.0 → v1.1）。
5. **文件位置**：将文档写入用户工作区根目录的 `ARCHITECTURE.md` 文件。
6. **不可遗漏**：检查所有节是否有内容，没有内容的节标明 "暂无" 或标注 `[若无可省略]` 后省略该节。
